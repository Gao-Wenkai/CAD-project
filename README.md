# AI 辅助编程大作业协作规范

本仓库用于完成“基于 MFC 的工程 CAD 项目”大作业。项目目标是模仿 AutoCAD 2022 的界面与基础二维绘图功能，在 AI 辅助下实现一个自主 CAD 软件原型。

由于本项目采用多人协作和 AI 辅助开发，所有成员在写代码前应先明确模块边界、统一风格，并通过分支提交代码，避免直接在主分支上修改。

## 1. 协作流程

### 1.1 分支规范

禁止直接在 `main` / `master` 分支上开发。每次开始新任务前，请先从最新主分支创建自己的功能分支。

推荐分支命名：

```text
feature/功能名-姓名
ui/界面模块-姓名
fix/问题描述-姓名
test/测试内容-姓名
docs/文档内容-姓名
```

示例：

```text
feature/line-circle-gaowenqi
ui-toolbar-zhangsan
fix-selection-bug-lisi
test-drawing-tools-wangwu
```

基本流程：

```bash
git pull
git checkout -b feature/your-task-name

# 完成代码后
git add .
git commit -m "feat: add line drawing tool"
git push origin feature/your-task-name
```

代码合并前应先发起 Pull Request / Merge Request，由至少一名同学检查后再合并。

### 1.2 提交信息规范

提交信息建议使用以下格式：

```text
类型: 简短说明
```

常用类型：

```text
feat: 新功能
fix: 修复 bug
ui: 界面调整
test: 测试相关
docs: 文档修改
refactor: 代码重构
chore: 构建、配置、杂项修改
```

示例：

```text
feat: add rectangle drawing command
fix: correct mouse drag coordinate conversion
ui: add toolbar button for circle tool
docs: update branch workflow
```

## 2. 代码与注释规范

### 2.1 基本原则

1. 每个功能模块应先规划再编码，避免边写边随意扩展全局变量或重复逻辑。
2. 尽量保持一个文件只负责一类职责，例如绘图命令、图元数据、界面交互、文件读写等不要混在一起。
3. AI 生成代码必须先读懂再提交，不能直接粘贴无法解释的代码。
4. 修改他人代码前，先确认它的设计意图；如果必须大改，应在群里说明原因。
5. 不提交临时文件、编译产物、个人 IDE 缓存和无关压缩包。

### 2.2 注释格式

为方便多人维护，统一采用以下注释风格。

文件头注释：

```cpp
// File: DrawCommand.cpp
// Purpose: Implement basic drawing commands for CAD entities.
// Author: 姓名
// Created: 2026-05-14
```

函数注释：

```cpp
// 功能: 根据鼠标输入创建直线图元
// 参数:
//   startPoint - 直线起点
//   endPoint   - 直线终点
// 返回: 创建成功返回 true，否则返回 false
bool CreateLine(const CPoint& startPoint, const CPoint& endPoint);
```

复杂逻辑注释：

```cpp
// 将屏幕坐标转换为绘图坐标，保证缩放和平移后图元位置一致。
```

注意：

1. 注释解释“为什么这样做”和“这段逻辑负责什么”，不要逐行翻译代码。
2. AI 生成的复杂代码必须补充人工可读的说明。
3. 未完成内容统一用 `TODO(姓名): 说明` 标记。

示例：

```cpp
// TODO(张三): 补充圆弧绘制时的角度约束逻辑。
```

## 3. AI / Vibe Coding 使用约定

本项目允许使用 AI 辅助编码，但需要保持代码结构可控。

### 3.1 使用 AI 前

在让 AI 写代码前，应先明确：

1. 当前模块负责什么。
2. 输入和输出是什么。
3. 是否会影响其他同学的模块。
4. 需要遵守哪些已有类名、函数名和文件结构。

建议提示词格式：

```text
请基于当前 MFC 项目结构，实现 [具体功能]。
要求：
1. 不修改无关文件。
2. 保持现有命名风格。
3. 添加必要注释。
4. 说明改动了哪些文件。
```

### 3.2 使用 AI 后

AI 生成代码后必须检查：

1. 是否能编译。
2. 是否破坏已有功能。
3. 是否引入重复类、重复函数或不必要的新框架。
4. 是否有难以理解的长函数。
5. 是否补充了必要注释。

## 4. 项目分工

初步将成员分为三组，后续可根据实际进度调整。

### 4.1 功能实现组

主要负责 CAD 核心功能。

建议任务：

1. 基础图元：直线、矩形、圆、椭圆、多段线等。
2. 鼠标交互：点击、拖拽、预览、确认绘制。
3. 图元管理：选择、删除、移动、撤销/重做。
4. 坐标系统：屏幕坐标与绘图坐标转换。
5. 文件保存与加载，如有时间可实现简单自定义格式。

### 4.2 界面设计组

主要负责界面布局和用户操作体验。

建议任务：

1. 模仿 AutoCAD 风格设计菜单栏、工具栏、绘图区和状态栏。
2. 为常用绘图工具添加按钮和快捷入口。
3. 显示当前工具、鼠标坐标、绘图状态等信息。
4. 优化窗口缩放、控件排列和基础视觉一致性。
5. 准备汇报 PPT 中的界面展示材料。

### 4.3 测试捉虫组

主要负责发现问题、复现问题和验收功能。

建议任务：

1. 制定基础测试清单。
2. 测试每个绘图工具的正常流程和异常操作。
3. 记录 bug：出现步骤、预期结果、实际结果、截图。
4. 回归测试已修复问题。
5. 汇总项目 bug 情况，用于最终汇报。

Bug 记录建议格式：

```text
标题:
发现人:
所属模块:
复现步骤:
预期结果:
实际结果:
截图或说明:
当前状态: open / fixed / verified
```

## 5. 阶段计划

### 阶段一：框架确认与任务拆分

目标：

1. 确认已有同学搭建的 MFC 项目框架。
2. 明确目录结构和主要类职责。
3. 将任务拆成可独立开发的小模块。
4. 每个成员认领任务并创建分支。

产出：

1. 初始可运行项目。
2. 模块分工表。
3. 第一版任务列表。

### 阶段二：核心功能开发

目标：

1. 实现基础二维图元绘制。
2. 完成主要鼠标交互流程。
3. 初步实现界面工具入口。
4. 保证项目可以稳定编译运行。

产出：

1. 可演示的基础 CAD 原型。
2. 主要绘图功能说明。
3. 初步测试记录。

### 阶段三：界面完善与功能整合

目标：

1. 优化界面布局和操作体验。
2. 合并各功能分支。
3. 修复模块整合后的冲突和 bug。
4. 准备汇报演示流程。

产出：

1. 较完整的项目版本。
2. 演示用测试图形。
3. PPT 初稿。

### 阶段四：测试、修复与提交

目标：

1. 系统测试所有核心功能。
2. 修复影响演示和评分的主要 bug。
3. 整理源码、说明文档和 PPT。
4. 打包最终提交材料。

产出：

1. 最终源码。
2. 界面和功能说明 PPT。
3. 提示词或 AI 使用记录汇总。
4. bug 修复记录。

## 6. 推荐目录结构

当前项目框架尚未完全确定，后续可根据已有 MFC 框架调整。建议保持大致结构清晰：

```text
/
├── README.md
├── docs/               # 文档、分工、测试记录、汇报材料
├── src/                # 主要源代码，具体结构按 MFC 工程调整
├── assets/             # 图标、截图、示例文件等资源
└── tests/              # 测试记录或辅助测试代码
```

如果已有同学的框架与此不同，以可运行框架为准，但应尽量保持文档、资源、测试记录有固定位置。

## 7. 合并前检查清单

提交合并请求前，请确认：

```text
[ ] 已从最新主分支创建独立分支
[ ] 项目可以正常编译
[ ] 功能经过基本手动测试
[ ] 没有提交编译产物和个人缓存文件
[ ] 新增复杂逻辑已有必要注释
[ ] 修改内容只涉及本次任务范围
[ ] 已在提交信息中说明主要改动
```

## 8. 当前优先级

1. 等待并确认初始 MFC 框架。
2. 整理项目目录和基础构建方式。
3. 确定第一批功能：直线、矩形、圆、选择、删除。
4. 界面组先规划工具栏和绘图区布局。
5. 测试组同步建立 bug 记录模板和验收清单。
